# Double authentification (2FA) par application TOTP — Design

Date : 2026-06-25
Statut : validé (brainstorming) — prêt pour le plan d'implémentation

## Objectif

Permettre à un membre d'activer la **2FA via application d'authentification** (TOTP : Google
Authenticator, Aegis, 1Password…). À la connexion, après le mot de passe, un code à 6 chiffres (ou
un **code de secours**) est requis. Réservé à la sécurité du compte ; l'UI Réglages → Sécurité a
déjà un emplacement (« 2FA — Bientôt »).

## Décisions

- Inclut des **codes de secours** (8, usage unique) pour éviter le verrouillage en cas de perte du
  téléphone.
- Secret TOTP **chiffré au repos** (AES-256-GCM) ; clé **dérivée de `JWT_SECRET`** (`sha256`) → aucune
  nouvelle variable d'environnement.
- `authenticateUser` **inchangé** : la logique 2FA vit dans la route de login.

## 1. Modèle de données

`src/server/domain/entities.ts` — ajouter à `User` :

```ts
  twoFactorEnabled: boolean;            // existe déjà
  twoFactorSecret: string | null;       // secret TOTP chiffré (null si non configuré)
  twoFactorBackupCodes: string[];       // hash sha256 des codes de secours non consommés
```

Tous les points de construction d'un `User` reçoivent les valeurs par défaut
`twoFactorSecret: null`, `twoFactorBackupCodes: []` : `registerUser`, les mappers Mongo
(`fromUserDoc`/`toUserDoc`), et les helpers `user(...)` des tests. À l'**activation**, le secret est
posé mais `twoFactorEnabled` reste `false` jusqu'à confirmation d'un code.

## 2. Ports & adaptateurs (hexagonal)

`src/server/ports/security.ts` :

```ts
export interface Totp {
  generateSecret(): string;
  keyUri(secret: string, account: string, issuer: string): string;
  verify(code: string, secret: string): boolean;
}
export interface Crypto {
  encrypt(plain: string): string;        // → "iv:tag:ciphertext" base64
  decrypt(cipher: string): string;       // lève si altéré/illisible
  hash(value: string): string;           // sha256 hex (pour codes de secours haute entropie)
}
export interface TokenService {
  issue(payload: { userId: string }): Promise<string>;              // existant (session 7j)
  verify(token: string): Promise<{ userId: string } | null>;        // existant
  issueChallenge(userId: string): Promise<string>;                  // JWT court (5 min, scope "2fa")
  verifyChallenge(token: string): Promise<{ userId: string } | null>;
}
```

Adaptateurs (`src/server/adapters/security/`) :
- `otplib-totp.ts` — `OtplibTotp implements Totp` (otplib `authenticator`).
- `node-crypto.ts` — `NodeCrypto implements Crypto` : AES-256-GCM (clé = `createHash("sha256").update(jwtSecret).digest()`), IV aléatoire 12 octets par message ; `hash` = `sha256` hex.
- `jwt-token-service.ts` (modifié) — ajoute `issueChallenge` (claim `scope:"2fa"`, exp `5m`) et
  `verifyChallenge` (rejette si `scope !== "2fa"`). Le `verify` de session rejette les tokens
  `scope:"2fa"` (un challenge ne doit jamais valoir session).

Container (`src/server/container.ts`) : `Deps` gagne `totp: Totp` et `crypto: Crypto`.
- `makeInMemoryDeps` : vrais adaptateurs (`new OtplibTotp()`, `new NodeCrypto("test-secret-…")`) pour
  des tests fidèles.
- `getDeps` : `new OtplibTotp()`, `new NodeCrypto(env.JWT_SECRET)`.

Dépendances npm : `otplib`, `qrcode` (+ `@types/qrcode`), serveur uniquement.

## 3. Application — `src/server/application/two-factor.ts`

```ts
const ISSUER = "vulu";
const BACKUP_CODE_COUNT = 8;

// Génère un secret, le chiffre, le stocke (enabled inchangé/false), renvoie de quoi enrôler.
export async function beginTwoFactorSetup(deps, userId): Promise<{ keyUri: string; secret: string; qrSvg: string }>;

// Vérifie le code TOTP contre le secret en attente ; si OK → backup codes (clairs, une fois),
// hashés et stockés, twoFactorEnabled = true. Renvoie { backupCodes }.
export async function confirmTwoFactorSetup(deps, userId, code: string): Promise<{ backupCodes: string[] }>;

// Vérifie le mot de passe, efface secret + codes, twoFactorEnabled = false.
export async function disableTwoFactor(deps, userId, password: string): Promise<void>;

// Login étape 2 : vérifie le challenge, puis le code TOTP OU un code de secours (consommé).
// Renvoie un token de session.
export async function completeTwoFactorLogin(deps, challenge: string, code: string): Promise<{ token: string }>;
```

Règles :
- `beginTwoFactorSetup` : `secret = deps.totp.generateSecret()` ; `keyUri = deps.totp.keyUri(secret, user.username, ISSUER)` ; stocke `twoFactorSecret = deps.crypto.encrypt(secret)` ; `qrSvg = await qrSvgFromUri(keyUri)` (helper `qrcode`). Ne touche pas `twoFactorEnabled`.
- `confirmTwoFactorSetup` : refuse si `twoFactorSecret` est null (`ValidationError`). `secret = deps.crypto.decrypt(...)` ; si `!deps.totp.verify(code, secret)` → `AuthError("Code invalide")`. Sinon génère `BACKUP_CODE_COUNT` codes (format `XXXX-XXXX`, alphanumérique majuscule sans ambiguïté), stocke `twoFactorBackupCodes = codes.map(deps.crypto.hash)`, `twoFactorEnabled = true`. Renvoie les codes **en clair** (affichés une seule fois).
- `disableTwoFactor` : `AuthError` si mot de passe faux ; sinon `twoFactorSecret = null`, `twoFactorBackupCodes = []`, `twoFactorEnabled = false`.
- `completeTwoFactorLogin` : `verifyChallenge` → `userId` (sinon `AuthError`) ; charge l'utilisateur ; si `deps.totp.verify(code, decrypt(secret))` → OK ; sinon si `deps.crypto.hash(normalize(code))` ∈ `twoFactorBackupCodes` → retire ce hash, persiste, OK ; sinon `AuthError`. Émet et renvoie `deps.tokens.issue({ userId })`.

## 4. Routes HTTP

- `POST /api/auth/login` (modifié, `src/app/api/auth/login/route.ts`) : après `authenticateUser`, si
  `user.twoFactorEnabled` → `challenge = await deps.tokens.issueChallenge(user.id)`,
  **pas de cookie**, `json({ twoFactorRequired: true, challenge })`. Sinon : `setSessionCookie(token)`
  + `json({ user, reactivated })` (comportement actuel).
- `POST /api/auth/login/2fa` (nouveau) : rate-limited (`authLimiter`), `{ challenge, code }`
  (`twoFactorLoginSchema`) → `completeTwoFactorLogin` → `setSessionCookie(token)` → `json({ ok: true })`.
- `POST /api/me/2fa/setup` (nouveau) : `requireUser` → `beginTwoFactorSetup` → `{ keyUri, secret, qrSvg }`.
- `POST /api/me/2fa/confirm` (nouveau) : `requireUser`, `{ code }` (`totpCodeSchema`) →
  `confirmTwoFactorSetup` → `{ backupCodes }`.
- `POST /api/me/2fa/disable` (nouveau) : `requireUser`, `{ password }` → `disableTwoFactor` → `{ ok: true }`.

Schémas (`src/lib/zod-schemas.ts`) :
```ts
export const totpCodeSchema = z.object({ code: z.string().min(6).max(12) });
export const twoFactorLoginSchema = z.object({ challenge: z.string().min(10), code: z.string().min(6).max(12) });
export const disable2faSchema = z.object({ password: z.string().min(1) });
```
(Le code accepte 6 chiffres TOTP **ou** un code de secours `XXXX-XXXX` ; la normalisation —
majuscules, retrait des tirets/espaces — est faite côté application.)

## 5. UI

- **Réglages → Sécurité** (`src/app/parametres/SettingsClient.tsx`) : remplacer le badge « Bientôt ».
  - Désactivée → bouton **Activer** → `setup` → affiche le **QR** (`qrSvg`), le secret en clair (repli
    saisie manuelle), un champ code → **Confirmer** (`confirm`) → écran **codes de secours**
    (copier / télécharger .txt) → état « Activée ».
  - Activée → bouton **Désactiver** → demande le **mot de passe** → `disable`.
  - Le composant 2FA est isolé dans `src/components/TwoFactorSettings.tsx` (`"use client"`) pour ne pas
    alourdir `SettingsClient`.
- **Login** (`src/app/(auth)/login/page.tsx`) : si la réponse contient `twoFactorRequired`, basculer
  sur un écran « Code de vérification » (6 chiffres) + lien « Utiliser un code de secours » →
  `POST /api/auth/login/2fa` avec le `challenge` mémorisé → redirection comme un login normal.

## 6. Tests

- **Application** `tests/application/two-factor.test.ts` (`makeInMemoryDeps`) :
  - `beginTwoFactorSetup` : `twoFactorSecret` non null & **différent du secret clair** (chiffré),
    `twoFactorEnabled` reste `false`, `keyUri` contient `otpauth://totp/` et l'issuer.
  - `confirmTwoFactorSetup` : code TOTP **généré via otplib** depuis le secret → `enabled=true`,
    8 codes renvoyés ; code erroné → `AuthError`, reste désactivée.
  - `completeTwoFactorLogin` : challenge valide + TOTP courant → token ; **code de secours** accepté
    puis **refusé à la 2ᵉ utilisation** ; challenge invalide/expiré → `AuthError`.
  - `disableTwoFactor` : bon mot de passe → tout effacé ; mauvais → `AuthError`.
- **Adaptateurs** :
  - `tests/adapters/node-crypto.test.ts` : `encrypt`→`decrypt` round-trip ; message altéré → throw ;
    `hash` déterministe.
  - `tests/adapters/otplib-totp.test.ts` : `verify(generate(secret), secret)` vrai ; mauvais code faux.
  - `jwt-token-service` : `verifyChallenge` accepte un challenge, `verify` (session) **rejette** un
    challenge `scope:"2fa"` et inversement.
- **zod** `tests/lib/zod-schemas.test.ts` : `totpCodeSchema`, `twoFactorLoginSchema`, `disable2faSchema`.
- Tests existants (`authenticate-user`, `account`, `auth`, mappers, `register-user`) : adaptés pour les
  nouveaux champs `User` par défaut.

## 7. Sécurité — points de vigilance

- Secret TOTP **jamais** renvoyé après l'activation ; chiffré en base (un dump DB ne donne pas les
  secrets sans `JWT_SECRET`).
- Codes de secours stockés **hachés**, affichés une seule fois, **usage unique** (retirés à la conso).
- Le challenge de login est un JWT **court (5 min)**, `scope:"2fa"`, sans valeur de session.
- `/api/auth/login/2fa` est **rate-limité** (comme le login) pour borner le brute-force du code.
- Aucun secret en clair dans les logs.
